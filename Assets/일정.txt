ActionMenuSO는 실행 트리거 역할만 수행
실제 기능 실행 주체는 각 Entity (예: BedEntity.Sleep)
구조 방향 확정:
BaseEntity
├─ 개별Entity (침대, 상자, 게시판, 책상 등)
├─ ItemEntity (Weapon 등)
└─ ActorEntity (Player, Monster)
설계 원칙:
SO는 수치/고정 데이터/연출 정보만 담당
Entity는 실제 게임 로직 실행 담당
ActionMenuSO는 Entity의 기능을 호출하는 트리거 역할
Managers를 통해 전역 시스템 접근 (Game, UI, Sound, Area, Save 등)

기획의 구조적 문제로 방식 구상.
A안. 텍스트로 상호작용 오브젝트 표현
- 장점: 기존 게임들과 다른 방식으로 게임 표현 가능, 몇 개월전에 Text-Animator에셋 분석 및 구현된 일부 로직 존재
- 단점: 끊임없이 충돌되는 기획, 직관적이지 않아 기획 부분 보강 필요 (예를 들어 텍스트 애니메이션 표현 방식)

B안. 스프라이트로 상호작용 오브젝트 표현
- 장점: A안과 C안의 장점을 합침, A안보다 직관적으로 표현
- 단점: A안과 C안의 단점도 합침, 여전히 기획 필요

C안. 스프라이트+그리드 시스템 활용
- 장점: 
- 단점: 기존에 널린 보편적인 게임과 같은 게임

C안 = 포트폴리오용으로 제격, A안 = 한번 만들고 싶었던 게임을 구현하면서 포트폴리오로 활용하기 위해 선택 했음, 그러나 애매한 시간과 불확실한 기획.

기획 → 구조 설계 → 개발 이 흐름 자체가 포트폴리오용으로 맞지 않음.
사유: 기획 완료 -> 설계 -> 개발을 진행하며 기획적으로 문제인 부분 발생. 다시 기획 -> 다시 설계 -> 다시 개발이 반복.
해결 방안: 큰 틀 안에서 게임 흐름이 가능하도록 구현

SRP (Single Responsibility Principle, 단일 책임 원칙): 
클래스는 단 하나의 책임(기능)만 가져야 하며, 변경해야 할 이유는 오직 하나여야 합니다.

OCP (Open/Closed Principle, 개방-폐쇄 원칙): 
소프트웨어 요소는 확장에는 열려 있으나, 변경에는 닫혀 있어야 합니다. 기존 코드를 변경하지 않고 기능을 추가할 수 있어야 합니다.

LSP (Liskov Substitution Principle, 리스코프 치환 원칙): 
프로그램의 객체는 프로그램의 정확성을 깨뜨리지 않으면서 하위 타입의 인스턴스로 바꿀 수 있어야 합니다 (자식 클래스는 부모 클래스의 책임을 준수해야 함).

ISP (Interface Segregation Principle, 인터페이스 분리 원칙): 
특정 클라이언트를 위한 인터페이스 여러 개가 범용 인터페이스 하나보다 낫습니다. 즉, 자신이 사용하지 않는 메서드에 의존하지 않아야 합니다.

DIP (Dependency Inversion Principle, 의존관계 역전 원칙): 
구체화된 클래스(구현체)보다 추상화된 클래스(인터페이스)에 의존해야 합니다. 의존성 주입을 통해 결합도를 낮추는 것을 의미합니다.
